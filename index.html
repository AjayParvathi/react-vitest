How to Write Test Cases Using Vitest

Everything a React beginner needs to know — from first test to reading coverage reports — written in plain, simple English with no confusing jargon.

vitest @testing-library/react react + vite beginner friendly

01 What is Testing?

Imagine you build a light switch. Before giving it to someone, you flip it up and down a few times to check it actually turns the light on and off. That's testing — checking your code does what it's supposed to do.

A test is just more code that runs your real code and checks whether the result is correct. If it's wrong, the test fails and tells you exactly where.

Simple definitionA test is code that checks your code. Write it once, run it any time — especially after changes — to instantly know if everything still works.

Four types of tests

Unit tests

  • Test one small piece at a time
  • A single function or hook
  • Very fast
  • Most common — start here

Integration tests

  • Test multiple pieces together
  • Component + its hook
  • Closer to real usage
  • Also very important

End-to-end (E2E)

  • Tests full app in real browser
  • Clicks like a real user
  • Slowest to run
  • Playwright or Cypress

Snapshot tests

  • Saves a "photo" of UI output
  • Fails if UI changes unexpectedly
  • Good for catching regressions
  • Covered in section 15

02 Why Write Tests?

You might think "I can just run the app and click around." That's manual testing — and it breaks down fast:

  1. It gets too slow as your app grows

    With 50 features, manually checking everything after every change takes forever.

  2. You forget to check things

    You fix one thing, accidentally break another, and don't notice until a user complains.

  3. You become afraid to change code

    Without tests, every refactor feels risky. With tests, you can boldly make changes and know in seconds if something broke.

  4. Onboarding new teammates is harder

    Tests act as living documentation that shows exactly how the code is supposed to behave.

The golden benefitTests give you confidence. You change something, run tests, all green — and you ship without fear.

04 Anatomy of a Test

full structure
describe('Button component', () => {   // groups related tests

  beforeEach(() => {                  // runs before EVERY test below
    vi.clearAllMocks();
  });

  afterEach(() => {                   // runs after EVERY test below
    // cleanup if needed
  });

  it('should show the correct label', () => {

    // ARRANGE — set up what you need
    render(<Button label="Save" />);

    // ACT — do the thing you want to test
    const button = screen.getByRole('button');

    // ASSERT — check the result
    expect(button).toHaveTextContent('Save');

  });

});
PartWhat it meansExample
ArrangeSet up everything needed before the testRender a component, mock an API
ActDo the thing you want to testClick a button, type in a field
AssertCheck if the result is what you expectedexpect(x).toBe('something')
Lifecycle hooksbeforeAll — runs once before all tests in the block. afterAll — runs once after all tests. beforeEach — runs before every single test. afterEach — runs after every single test. You can nest these inside describe blocks.

05 Matchers — The expect() Function

A matcher checks a value. Pattern: expect(actualValue).matcherName(expectedValue)

General value matchers

MatcherChecks that…
.toBe(val)Exact equality — like === (use for primitives: strings, numbers, booleans)
.toEqual(val)Deep equality — use for objects and arrays (compares contents)
.toStrictEqual(val)Like toEqual but also checks object types (stricter)
.toBeTruthy()Value is truthy (anything that isn't false, null, 0, "", undefined)
.toBeFalsy()Value is falsy
.toBeNull()Value is exactly null
.toBeUndefined()Value is undefined
.toBeDefined()Value is not undefined
.toBeGreaterThan(n)Number is greater than n
.toBeLessThan(n)Number is less than n
.toContain(item)Array includes item, or string includes substring
.toHaveLength(n)Array or string has exactly n items/characters
.toMatch(/regex/)String matches a regular expression
.toThrow()Function throws an error when called
.toThrow('message')Function throws an error with that specific message

Mock function matchers

MatcherChecks that…
.toHaveBeenCalled()The mock function was called at least once
.toHaveBeenCalledTimes(n)The mock was called exactly n times
.toHaveBeenCalledWith(...args)The mock was called with those exact arguments
.toHaveBeenLastCalledWith(...args)The most recent call used those arguments
.toHaveReturnedWith(val)The mock returned that value

DOM matchers (from jest-dom)

MatcherChecks that…
.toBeInTheDocument()Element exists in the fake DOM
.toHaveTextContent('...')Element's visible text matches
.toHaveValue('...')Input/select/textarea has that value
.toBeDisabled()Button or input is disabled
.toBeEnabled()Button or input is enabled
.toBeVisible()Element is visible to the user
.toHaveClass('name')Element has a specific CSS class
.toHaveAttribute('attr', 'val')Element has an HTML attribute with that value
.toHaveFocus()Element currently has keyboard focus
.toBeChecked()Checkbox or radio is checked
.toBeEmptyDOMElement()Element has no visible content
The .not modifierPut .not before any matcher to flip it: expect(x).not.toBeNull(), expect(fn).not.toHaveBeenCalled(), expect(el).not.toBeInTheDocument()

06 Testing Custom Hooks

Custom hooks can't be called directly in tests — React requires hooks to run inside a component. renderHook creates a hidden mini component for you and runs the hook inside it.

useCounter.test.js
import { renderHook, act } from '@testing-library/react';
import { useCounter } from './useCounter';

describe('useCounter', () => {

  it('should start with count = 0', () => {
    const { result } = renderHook(() => useCounter());
    // result.current = everything the hook returns
    expect(result.current.count).toBe(0);
  });

  it('should increment when increment() is called', () => {
    const { result } = renderHook(() => useCounter());

    // act() = "finish processing all state changes before I check"
    act(() => {
      result.current.increment();
    });

    expect(result.current.count).toBe(1);
  });

});
Why act()?React batches state updates. Without act(), your assertions run before React has finished updating state — so you check old values. act() says "wait for React to finish, then continue."

07 Testing Components

Use render() to mount a component into a fake DOM, then screen to find and interact with elements.

Button.test.jsx
import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import { Button } from './Button';

describe('Button', () => {

  it('should display the label', () => {
    render(<Button label="Save" />);
    expect(screen.getByText('Save')).toBeInTheDocument();
  });

  it('should call onClick when clicked', async () => {
    const handleClick = vi.fn();
    render(<Button label="Save" onClick={handleClick} />);

    await userEvent.click(screen.getByRole('button'));

    expect(handleClick).toHaveBeenCalledTimes(1);
  });

  it('should be disabled when disabled prop is true', () => {
    render(<Button label="Save" disabled={true} />);
    expect(screen.getByRole('button')).toBeDisabled();
  });

});

userEvent vs fireEvent

You might see both in tutorials. Here's the difference:

userEvent (recommended)

  • Simulates real browser behavior
  • Fires all the right events in order (mousedown, mouseup, click...)
  • Use await userEvent.click()
  • More realistic, catches more bugs

fireEvent (older)

  • Fires just one event directly
  • Simpler but less realistic
  • Use only if userEvent has issues
  • fireEvent.click(button) — no await

08 The screen Object

screen is your eyes into the fake DOM. After render(), use it to find elements the way a real user would.

Three flavours of every query

FlavourIf element missing…Use when…
getBy…Throws error immediatelyElement MUST be on screen right now
queryBy…Returns null (no error)Checking something is NOT there
findBy…Waits, then throws if still missingElement appears after API call or delay (async)

The query methods (how to find elements)

MethodFinds by…Recommended?
getByRole('button', {name: '…'})HTML role + accessible nameYes — best choice
getByLabelText('Name')The text of a label elementYes — best for form inputs
getByText('Submit')Visible text on the pageYes — simple and readable
getByPlaceholderText('…')Input placeholder attributeSometimes
getByAltText('…')Image alt attributeFor images only
getByDisplayValue('…')Current value of an input or selectOccasionally
getByTitle('…')title attributeRarely
getByTestId('my-id')data-testid="…" attributeLast resort only

getAllBy — when multiple elements match

javascript
// getAllBy returns an array — use when you expect more than one match
const items = screen.getAllByRole('listitem');
expect(items).toHaveLength(3);

// Pair: getBy / queryBy / findBy = single element
// Pair: getAllBy / queryAllBy / findAllBy = multiple elements
Why getByRole is the best choiceIt finds elements the same way screen readers and accessibility tools do. Good tests and good accessibility go hand in hand — if your test can find the button, so can a keyboard user.

09 Mocking

A mock is a fake replacement. You mock things you don't want to really run during tests — like API servers, navigation, or toast popups.

vi.fn() — mock a single function

javascript
const mockSave = vi.fn();  // fake function that does nothing but records calls

render(<Form onSave={mockSave} />);

expect(mockSave).toHaveBeenCalled();
expect(mockSave).toHaveBeenCalledWith({ name: 'Alice' });
expect(mockSave).toHaveBeenCalledTimes(1);

vi.mock() — mock an entire module

javascript
// Replace the whole react-router module with a fake version
const mockNavigate = vi.fn();

vi.mock('react-router', () => ({
  useNavigate: () => mockNavigate,
}));

// Import AFTER vi.mock() — you get the fake version
import { useNavigate } from 'react-router';

Making mocks return values

javascript
const mockFn = vi.fn();

mockFn.mockReturnValue('hello');          // returns 'hello' every time
mockFn.mockReturnValueOnce('first');       // returns 'first' only on the first call
mockFn.mockResolvedValue({ ok: true });    // returns a resolved Promise
mockFn.mockRejectedValue(new Error('fail')); // returns a rejected Promise
mockFn.mockImplementation((x) => x * 2);   // custom logic

Clearing mocks between tests

javascript
beforeEach(() => {
  vi.clearAllMocks();  // resets call history + return values for ALL mocks
});

// Or for a specific mock:
mockFn.mockClear();   // resets call history only
mockFn.mockReset();   // resets call history + return values
mockFn.mockRestore(); // restores to original (only works with spyOn)

10 vi.spyOn() — Watch Without Replacing New

This is something beginners often miss. vi.fn() creates a completely fake function. vi.spyOn() wraps an existing real function so you can watch it — but it still runs the original code unless you tell it otherwise.

vi.fn()

  • Creates a brand new fake function
  • Does nothing by default
  • Use for callbacks passed as props
  • Use when you don't want real code to run

vi.spyOn()

  • Wraps an existing real function
  • Real code still runs (unless mocked)
  • Use to watch methods on objects/modules
  • Must call .mockRestore() in afterEach
javascript
// Example: spy on console.error to check if errors are logged
it('should log an error on failure', () => {
  const spy = vi.spyOn(console, 'error').mockImplementation(() => {}); // silence it

  render(<BrokenComponent />);

  expect(spy).toHaveBeenCalled();
  spy.mockRestore(); // IMPORTANT: restore console.error after the test
});

// Example: spy on a module method but keep real behavior
import * as utils from './utils';

it('should call formatDate', () => {
  const spy = vi.spyOn(utils, 'formatDate'); // real function still runs!

  render(<DateDisplay />);

  expect(spy).toHaveBeenCalled();
});
Rule of thumbUse vi.fn() when you don't want real code to run at all. Use vi.spyOn() when you want to watch a real function and still let it run, or when you need to temporarily override a method on an existing object.

11 Async Testing

async/await in tests

javascript
it('should save and show success message', async () => {
  mockApi.mockResolvedValue({ success: true });
  render(<Form />);

  await userEvent.click(screen.getByRole('button', { name: 'Save' }));

  // findBy waits for the element to appear (up to 1 second by default)
  const msg = await screen.findByText('Saved successfully!');
  expect(msg).toBeInTheDocument();
});

waitFor — wait for any condition

javascript
import { render, screen, waitFor } from '@testing-library/react';

it('should call the API after button click', async () => {
  render(<Form />);
  await userEvent.click(screen.getByRole('button'));

  // keeps re-running the check until it passes (or times out)
  await waitFor(() => {
    expect(mockApi).toHaveBeenCalledTimes(1);
  });
});

act() with async hooks

javascript
// When a hook function is async (returns a Promise)
await act(async () => {
  await result.current.handleSubmit();
});

12 Testing with Providers New

Many components need a provider wrapper to work — React Context, Redux store, React Query, React Router. Without the wrapper, the component crashes with an error like "must be used within a Provider."

Option A — wrap inline in each test

javascript
import { QueryClient, QueryClientProvider } from '@tanstack/react-query';

it('should load user data', async () => {
  const queryClient = new QueryClient();

  render(
    <QueryClientProvider client={queryClient}>
      <UserProfile />
    </QueryClientProvider>
  );

  expect(await screen.findByText('John Doe')).toBeInTheDocument();
});

Option B — create a custom render helper (recommended)

Instead of wrapping in every single test, create a custom renderWithProviders helper once and use it everywhere:

src/test-utils.jsx — create this file once
import { render } from '@testing-library/react';
import { BrowserRouter } from 'react-router-dom';
import { QueryClient, QueryClientProvider } from '@tanstack/react-query';

function AllProviders({ children }) {
  const queryClient = new QueryClient({
    defaultOptions: { queries: { retry: false } }, // no retries in tests
  });

  return (
    <BrowserRouter>
      <QueryClientProvider client={queryClient}>
        {children}
      </QueryClientProvider>
    </BrowserRouter>
  );
}

// Custom render — wraps with all providers automatically
export function renderWithProviders(ui, options) {
  return render(ui, { wrapper: AllProviders, ...options });
}

// Re-export everything from testing-library so tests only need one import
export * from '@testing-library/react';
UserProfile.test.jsx — using the helper
// Import from YOUR helper, not from @testing-library/react
import { renderWithProviders, screen } from '../test-utils';

it('should show user name', async () => {
  renderWithProviders(<UserProfile />); // providers added automatically!
  expect(await screen.findByText('John Doe')).toBeInTheDocument();
});

Passing props to Context for testing

javascript
// If you use React Context, you can pass a specific test value
render(
  <UserContext.Provider value={{ name: 'Alice', role: 'admin' }}>
    <Dashboard />
  </UserContext.Provider>
);

13 rerender and cleanup New

rerender — change props during a test

Sometimes you need to test what happens when a component receives different props. Use rerender from the render return value:

javascript
it('should update label when prop changes', () => {
  // render returns a rerender function
  const { rerender } = render(<Button label="Save" />);

  expect(screen.getByText('Save')).toBeInTheDocument();

  // Change props — like the parent re-rendering with new values
  rerender(<Button label="Update" />);

  expect(screen.getByText('Update')).toBeInTheDocument();
  expect(screen.queryByText('Save')).not.toBeInTheDocument();
});

cleanup — automatic DOM clearing

After every test, React Testing Library automatically removes the rendered component from the fake DOM. This is called cleanup and it happens automatically — you don't need to do anything. It prevents one test's HTML from leaking into the next test.

Good to know — you don't need to call cleanup manuallyTesting Library calls cleanup() automatically after each test via an afterEach hook it sets up for you. You only need to call it manually if you're using a very old version or a non-standard setup.

unmount — removing the component mid-test

javascript
it('should clean up subscriptions on unmount', () => {
  const { unmount } = render(<Timer />);

  unmount(); // simulates the component being removed from the page

  // Check that cleanup happened (e.g. clearInterval was called)
  expect(mockClearInterval).toHaveBeenCalled();
});

14 skip and only New

Vitest gives you control over which tests run. Very useful when debugging one failing test in a large file.

javascript
// .skip — skip this test, it won't run at all
it.skip('this test is temporarily disabled', () => {
  // ...
});

// .only — run ONLY this test, skip all others in the file
it.only('focus on just this one test while debugging', () => {
  // ...
});

// .skip on describe — skip an entire group
describe.skip('this whole group is disabled', () => {
  // all tests inside are skipped
});

// .only on describe — run only this group
describe.only('debug just this group', () => {
  // only tests inside here run
});

// .todo — marks a test as something you plan to write later
it.todo('should handle network timeout');
// shows up in output as a reminder without failing
Warning: never commit .only to your codebaseIf you leave it.only in a file and push to production, your CI/CD pipeline will only run that one test and think everything else is fine. Always remove .only before committing.

15 Snapshot Testing New

A snapshot test takes a "photo" of your component's rendered HTML and saves it to a file. The next time you run tests, it compares the current output to the saved photo. If they're different, the test fails.

javascript
import { render } from '@testing-library/react';

it('should match snapshot', () => {
  const { container } = render(<Button label="Save" />);

  // First run: creates a snapshot file
  // Later runs: compares against saved snapshot
  expect(container).toMatchSnapshot();
});

When you intentionally change a component and the snapshot test fails (because the UI changed on purpose), you need to update the saved snapshot:

terminal
# Update all snapshots to match the new output
npx vitest run --update-snapshots
Be careful with snapshotsSnapshots are easy to write but can give false confidence. If you blindly update them every time they fail, you miss real bugs. Prefer explicit assertions (like getByText, toHaveValue) when possible — save snapshots for components that are very stable and rarely change.

16 Test Coverage New

Coverage tells you which lines of your real code are actually being executed by your tests. It's shown as a percentage.

Running the coverage report

terminal
# First install the v8 coverage provider
npm install -D @vitest/coverage-v8

# Then run
npm run test:coverage

Reading the output

The terminal shows a table like this:

coverage output
File              | % Stmts | % Branch | % Funcs | % Lines | Uncovered
------------------|---------|----------|---------|---------|----------
useCounter.js     |   100   |   100    |   100   |   100   |
Button.jsx        |    80   |    60    |   100   |    80   | 24, 31
Form.jsx          |    45   |    30    |    66   |    45   | 12-28, 44
ColumnWhat it means
% StmtsPercentage of statements (lines of code) that ran during tests
% BranchPercentage of if/else branches that were tested (both true and false)
% FuncsPercentage of functions that were called
% LinesSimilar to Stmts — percentage of lines executed
UncoveredThe exact line numbers that no test ever reached
Don't chase 100% coverage100% coverage sounds great but it can be misleading. A test can execute a line without actually checking that it does the right thing. Aim for ~70–80% coverage on important logic. Focus on testing behavior, not chasing numbers.

18 Common Patterns

Testing a form submission

javascript
it('should submit with typed values', async () => {
  const onSubmit = vi.fn();
  render(<LoginForm onSubmit={onSubmit} />);

  await userEvent.type(screen.getByLabelText('Email'), 'user@test.com');
  await userEvent.type(screen.getByLabelText('Password'), 'secret');
  await userEvent.click(screen.getByRole('button', { name: 'Log in' }));

  expect(onSubmit).toHaveBeenCalledWith({
    email: 'user@test.com',
    password: 'secret',
  });
});

Testing conditional rendering

javascript
// Check it SHOWS when condition is true
it('should show error when field is empty', async () => {
  render(<Form />);
  await userEvent.click(screen.getByRole('button', { name: 'Submit' }));
  expect(screen.getByText('Name is required')).toBeInTheDocument();
});

// Check it HIDES when condition is false (use queryBy, not getBy)
it('should NOT show error when field is filled', async () => {
  render(<Form />);
  await userEvent.type(screen.getByLabelText('Name'), 'Alice');
  await userEvent.click(screen.getByRole('button', { name: 'Submit' }));
  expect(screen.queryByText('Name is required')).not.toBeInTheDocument();
});

Testing a loading state

javascript
it('should show loading then data', async () => {
  mockFetch.mockResolvedValue({ name: 'Alice' });
  render(<UserProfile />);

  // Loading state should appear first
  expect(screen.getByText('Loading...')).toBeInTheDocument();

  // Wait for data to appear
  expect(await screen.findByText('Alice')).toBeInTheDocument();

  // Loading spinner should now be gone
  expect(screen.queryByText('Loading...')).not.toBeInTheDocument();
});

19 What NOT to Test New

This is one of the most important lessons. Writing unnecessary tests wastes time and makes your test suite brittle — tests that break every time you refactor even when nothing is actually wrong.

Don't test these things

WhatWhy not
Third-party libraries (React, Axios, Redux)They have their own tests. Trust them.
Implementation details (internal variable names, state structure)If you refactor the code but behavior stays the same, tests shouldn't break
CSS styles and visual designHard to test reliably. Use visual regression tools (Chromatic, Percy) instead
Simple prop drilling (passing a value down and rendering it)Not enough logic to warrant a test
Every single line of code100% coverage doesn't mean 100% correct — focus on meaningful behavior

Focus your tests on these instead

WhatWhy
Business logic in hooks and utilitiesThis is the brain of your app — must be correct
User interactions (click, type, submit)What actually matters to the user
Error states (empty, failed API, validation)Easy to forget manually, often where bugs hide
Conditional rendering (show/hide logic)Easy to break when code changes
API integration (with mocks)Confirm the right data is sent and responses are handled
The key question to ask yourselfBefore writing a test, ask: "If this breaks, will a user notice?" If yes — write the test. If no — skip it.

21 Tips and Common Mistakes

One test = one behaviorEach it() should test exactly one thing. If you're checking three things, split into three tests. Small focused tests are easy to read and easy to debug.
Test behavior, not implementationDon't test internal details like variable names. Test what the user sees. If you refactor but behavior stays the same, tests should still pass.
Write test names as sentences"should show error when email is invalid" — when this fails, you instantly know what broke without reading the test body.
Prefer getByRole over getByTestIdgetByRole reflects real accessibility — it's how screen readers navigate too. getByTestId is a last resort and tells you nothing about real behavior.
Never commit .only or .skip without a reason.only will cause your CI pipeline to only run that one test. .skip silently skips tests that might be catching real bugs.
Don't forget act() around state changes in hook testsWithout it, React will warn you and your assertions may check stale values.
Using getBy when the element might be absentgetBy throws if the element isn't found. When checking absence, use queryBy — it returns null without crashing.
Not clearing mocks between testsCall vi.clearAllMocks() in beforeEach. Without it, call counts from Test 1 bleed into Test 2.
Making tests too specific about textIf your test checks exact wording like "Please enter a valid email address", it breaks every time a designer tweaks the copy. Check for key words instead, or test the presence of an error element by role.
Start small, build upDon't try to test everything at once. Write one test, make it pass, write another. Momentum beats perfection.
